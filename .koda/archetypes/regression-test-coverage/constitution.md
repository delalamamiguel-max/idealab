# Constitution: Regression Test Coverage

## Purpose
Automate creation, execution, and verification of regression test suites to ensure previously validated functionality remains intact after code changes.

---

### I. Hard-Stop Rules (Non-Negotiable)
✘ No untested user journeys or critical workflows
✘ No regenerated test code without deterministic assertions
✘ No reliance on manual verification for regression scope
✘ No flaky tests admitted into the suite
✘ No test runs without version-controlled logs and reports

---

### II. Mandatory Patterns (Must Apply)
✔ Leverage a dedicated regression test agent to scaffold, update, and execute suites
✔ Validate all legacy bug fixes with targeted regression cases
✔ Cover boundary conditions, negative paths, and data-driven scenarios
✔ Gate merges with automated regression runs in CI/CD
✔ Track pass/fail history per test and associate with code revisions
✔ Keep test data and environment configuration under source control

---

### III. Preferred Patterns (Recommended)
➜ Prioritize high-risk components via impact analysis
➜ Use parameterized and data-provider patterns to reduce duplication
➜ Employ test doubles for unstable external dependencies
➜ Schedule nightly or on-demand full regression jobs
➜ Maintain dashboards showing regression trends and flaky test quarantine status
➜ Incorporate notifications for regressions tied to owning teams

---
